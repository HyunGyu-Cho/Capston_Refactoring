# Requirements

이 문서는 "서비스가 반드시 해야 하는 일"을 기능 요구사항으로 정리한 문서입니다.

## Auth
### AUTH-001 회원가입
- 사용자는 이메일/비밀번호로 가입할 수 있어야 합니다.
- 이메일은 중복되면 안 됩니다.
- 비밀번호는 반드시 해시(예: BCrypt)로 저장해야 합니다.
- 가입 시 기본 권한은 `USER`입니다.

### AUTH-002 로그인/토큰 발급
- 사용자는 이메일/비밀번호로 로그인할 수 있어야 합니다.
- 로그인 성공 시 Access Token, Refresh Token을 발급해야 합니다.
- Access Token 만료는 30분입니다.
- Access Token에는 최소 `memberId`, `roles`가 들어가야 합니다.

### AUTH-003 재발급/로그아웃
- 유효한 Refresh Token으로 Access 재발급이 가능해야 합니다.
- Refresh Token 만료는 7일입니다.
- 로그아웃 시 해당 Refresh Token은 즉시 무효 처리해야 합니다.
- 토큰 탈취 대응을 위해 Rotation 정책을 지원해야 합니다.

### AUTH-004 인가 기본 정책
- 보호 API는 인증되지 않은 요청을 거부해야 합니다.
- 수정/삭제는 본인 또는 관리자만 가능해야 합니다.
- 관리자 API는 `ADMIN`만 접근 가능해야 합니다.

## 추천 흐름
### ANALYSIS-001 인바디 분석
- 사용자는 인바디 입력 후 체형 분석을 요청할 수 있어야 합니다.
- 시스템은 최신 인바디 기준으로 분석 결과를 저장/조회해야 합니다.

### SURVEY-001 설문 기반 추천
- 사용자는 설문을 작성하고 운동/식단 추천을 요청할 수 있어야 합니다.
- 추천 결과는 이력으로 저장되어야 합니다.

### WORKOUT-001 운동 추천
- 설문 기반으로 운동 추천을 생성/저장/조회할 수 있어야 합니다.

### DIET-001 식단 추천
- 설문 기반으로 식단 추천을 생성/저장/조회할 수 있어야 합니다.

## 커뮤니티
### COMM-001 카테고리
- 카테고리: 자유, 운동, 식단, 사용후기, 상품

### COMM-002 게시글/댓글/반응/신고
- 게시글/댓글 작성, 좋아요/싫어요, 신고가 가능해야 합니다.
- 익명 작성은 화면에서만 익명이고 내부 식별은 유지해야 합니다.

### COMM-003 수정/삭제
- 작성자 본인은 수정/삭제할 수 있어야 합니다.
- 삭제는 소프트 삭제를 사용해야 합니다.

### COMM-004 신고 운영
- 신고 누적 기준을 넘으면 관리자 알림이 발생해야 합니다.
- 관리자 조치 이력을 남겨야 합니다.

## 마이페이지
### MY-001 대시보드 조회
- 최신 인바디/분석/추천/구매내역 등을 한 화면에서 조회할 수 있어야 합니다.
- 데이터가 없으면 빈 상태를 명확히 보여줘야 합니다.
